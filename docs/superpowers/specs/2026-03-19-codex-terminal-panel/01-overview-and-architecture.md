# 总体方案与架构

## 目标

把右侧从“面向消息的聊天面板”替换为“面向真实 CLI 的单终端面板”，让用户直接在软件内操作 Codex 终端。

## 方案选择

采用以下结构：

- 前端使用 `xterm.js` 渲染终端内容。
- Rust 后端使用 PTY 管理真实终端进程。
- 终端启动命令固定为当前项目目录下执行 `codex`。
- 左侧项目、样式、数据连接、文件管理保持现状，不改交互模型。

不采用以下方案：

- 不保留现有聊天输入框再叠加一个终端模式开关。
- 不做 shell 终端和 Codex 终端二选一。
- 不直接移植 `cc-pane` 源码，避免把多标签、多分屏、多工作区一起带入当前仓库。

## 模块边界

### 前端

- `TerminalPanel`：右侧主容器，负责状态条、按钮、终端挂载点。
- `terminal-services.ts`：封装 Tauri IPC。
- `terminal-state.ts`：管理终端运行态。
- `xterm-adapter.ts`：封装 `xterm.js`、fit、输入回传和尺寸同步。

### 后端

- `terminal_manager.rs`：创建和管理 PTY session。
- `terminal_commands.rs`：暴露创建、写入、缩放、关闭命令。
- `terminal_event_bridge.rs`：把 PTY 输出转成前端事件。

## 与现有系统的关系

- 终端工作目录仍取当前项目配置里的项目目录。
- 文件同步继续监听项目目录下的 `reportlets` 变化并执行同步。
- 左侧配置保存后，不强制重启终端，但必须显式提示“配置已变化，请重启终端”。

## 设计原则

- 不做静默 fallback，终端启动失败直接暴露错误。
- 不复用当前聊天会话状态对象，避免两套模型耦合。
- 单项目单终端，保持实现边界清晰。
