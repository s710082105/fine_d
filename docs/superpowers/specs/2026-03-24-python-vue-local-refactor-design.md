# Python + Vue 本地重构设计

## 1. 背景与问题

当前仓库已经积累了可运行能力，但整体形态存在三个根因问题：

1. 同一类能力在多个位置重复实现，尤其是文件同步与远程交互，导致任何改动都难以完整覆盖验证。
2. 运行主链路混用了 `Rust`、`Tauri`、`Node`、`Python`、`Java` 等多种技术，跨进程和跨语言调用过深，调试成本高。
3. 现有实现以 `macOS` 开发环境为中心，落到 `Windows` 后暴露出 shell、路径、编码、运行时依赖等兼容性问题。

本次重构不是在旧架构上继续修补，而是直接重建一套新的本地开发工具，旧仓库仅保留为能力参考和迁移素材来源。

## 2. 重构目标

本次重构固定为以下目标：

1. 产品形态改为 `Python + Vue` 前后端分离。
2. 产品定位是单机本地开发工具，而不是多人共享的 Web 平台。
3. 交互入口统一为浏览器，不再以桌面壳程序作为主运行时。
4. 提供仓库级一键安装脚本，完成依赖安装、环境准备和启动前检查。
5. `Codex` 保留为辅助入口，不作为系统主入口。
6. 模块边界按领域职责划分，不再按技术栈拼装功能。

## 3. 非目标

本次设计明确不包含以下方向：

1. 不兼容保留现有 `Tauri/Rust/Java` 内部接口和运行边界。
2. 不以多人协作、集中部署、任务中心化为首版目标。
3. 不优先保留分散的自动同步机制，例如多套 hook、旁路 CLI、自定义 watcher 并存。
4. 不把 `Codex` 聊天入口设计为唯一工作流主链路。

## 4. 总体方案

采用“本地服务 + 浏览器界面 + 领域模块 + 适配器层”的重构方案。

### 4.1 运行形态

系统运行时拆为两个部分：

1. 本地 `Python` 服务
   - 提供 HTTP API、任务编排、文件读写、同步执行、预览调度。
   - 承担所有业务能力和外部系统集成。
2. 浏览器界面
   - 使用 `Vue` 实现主交互界面。
   - 只通过 API 与后端通信，不直接操作本地文件和系统命令。

`Node` 仅作为前端开发和构建工具存在，不再参与业务运行链路。

### 4.2 方案选择结论

本次重构选择的不是“单体随写随跑”，也不是“插件化平台”，而是：

- 单机本地工具
- `FastAPI` 本地服务
- 明确领域模块
- 外部依赖集中在适配器层

该方案相比现状的收益：

1. 保持交付和安装简单，适合单机使用。
2. 通过模块边界和适配器隔离，把 `Windows/macOS` 差异收敛到局部。
3. 让同步、数据探测、预览等核心能力共享同一主链路，降低重复实现。

## 5. 业务模块划分

首版固定为 5 个核心业务模块和 1 个协作模块。

### 5.1 核心业务模块

#### `project`

职责：

- 管理项目配置
- 维护目录映射
- 维护运行状态
- 执行环境检查

边界：

- 只回答“项目怎么组织、配置是什么、路径在哪里”
- 不直接承担同步、预览、数据探测逻辑

#### `datasource`

职责：

- 管理 FineReport 远程连接
- 进行数据集探测
- 执行 SQL 试跑
- 扫描字段结构

边界：

- 只回答“数据从哪里来、字段结构是什么”
- 不直接负责报表文件操作和发布

#### `reportlet`

职责：

- 管理 `.cpt` / `.fvs` 模板与报表文件
- 提供文件树、创建、复制、读写等能力

边界：

- 只处理报表文件本身
- 不直接处理远程发布和预览

#### `sync`

职责：

- 负责同步发布
- 负责远端回读校验
- 负责同步状态记录

边界：

- 只处理“如何发出去并确认结果”
- 不再允许其它模块自己拼同步逻辑

#### `preview`

职责：

- 打开浏览器预览
- 进行页面校验
- 截图和结果确认

边界：

- 只处理“怎么看、怎么验证”
- 不直接处理项目配置和同步规则

### 5.2 协作模块

#### `assistant`

职责：

- 组装项目上下文
- 接收自然语言任务
- 调度核心业务模块能力

边界：

- `Codex` 只是该模块的一个外部适配器
- 系统主功能必须在没有 `Codex` 时仍然完整可用

## 6. Codex 定位

`Codex` 在新系统中固定为辅助入口，而不是主入口。

### 6.1 允许的作用

1. 基于项目上下文发起自然语言任务
2. 批量触发已有业务能力
3. 辅助生成建议、说明和操作编排

### 6.2 不允许的作用

1. 不允许成为唯一配置入口
2. 不允许绕开正式业务 API 直接操作底层适配器
3. 不允许让核心业务依赖 `Codex` 才能工作

### 6.3 设计要求

所有 `Codex` 相关集成都必须落在：

- `application/assistant`
- `adapters/ai/codex`

核心业务模块不直接依赖 `Codex`。

## 7. 分层架构

建议使用以下分层：

### 7.1 `domain`

放业务规则和核心模型：

- 项目配置约束
- 路径映射规则
- 同步状态机
- 预览结果判断规则

### 7.2 `application`

放用例编排：

- 组合多个领域模块完成一个完整动作
- 负责用例输入、输出、调用顺序和状态推进

### 7.3 `adapters`

集中处理一切外部系统：

- `ai/codex`
- `fine`
- `browser`
- `platform`
- `system`

### 7.4 `infra`

放运行基础设施：

- 配置读取
- 日志
- 本地状态存储
- 任务记录

### 7.5 `schemas`

统一 API 的请求和响应模型，避免 UI、脚本、AI 各自定义字段结构。

## 9. 安装与启动闭环

### 9.1 安装脚本

提供仓库级安装脚本：

- `scripts/install-macos.sh`
- `scripts/install-windows.ps1`

职责：

1. 检查并安装 `Python`
2. 检查并安装 `Node`
3. 安装前后端依赖
4. 准备本地虚拟环境
5. 安装浏览器自动化依赖
6. 初始化运行目录和缓存目录

安装脚本必须显式输出失败原因，不做 silent fallback。

### 9.2 启动脚本

提供平台启动脚本：

- `scripts/start-macos.sh`
- `scripts/start-windows.cmd`

职责：

1. 启动本地 `FastAPI`
2. 启动前端开发服务，或托管已打包的前端静态文件
3. 自动打开默认浏览器
4. 输出监听地址、日志路径、健康检查结果

### 9.3 环境诊断脚本

提供：

- `scripts/doctor-macos.sh`
- `scripts/doctor-windows.ps1`

职责：

1. 检查 `Python/Node` 版本
2. 检查浏览器驱动状态
3. 检查端口占用
4. 检查工作目录权限
5. 检查 FineReport 相关本地依赖

## 10. 同步模块重构原则

同步能力必须集中到 `sync` 模块，禁止多处重复实现。

### 10.1 统一正式入口

所有同步动作只能走固定用例：

- `sync_file`
- `sync_directory`
- `pull_remote_file`
- `publish_project`
- `verify_remote_state`

UI、脚本、`Codex` 都只能调用这些用例，不能各自拼装旁路流程。

### 10.2 决策与执行分离

- `application/sync` 负责同步决策：
  - 源文件
  - 目标路径
  - 覆盖策略
  - 回读校验策略
- `adapters/fine` 或 `adapters/system` 负责真正执行：
  - 写入
  - 读取
  - 删除

这样协议变化和业务规则变化不会互相缠绕。

### 10.3 同步状态机

每次同步统一使用以下状态：

- `pending`
- `running`
- `success`
- `failed`
- `verified`

前端、日志、脚本、AI 都读取同一套状态定义。

### 10.4 自动同步策略

首版建议只保留：

1. 手动同步
2. 发布整个项目

自动同步如果后续需要恢复，也必须作为 `sync` 模块内部的一个可开关策略统一实现，不能再出现多套 hook、watcher、按钮逻辑并存。

### 10.5 同步闭环

一次完整同步固定为：

1. 读取本地源
2. 计算目标路径
3. 执行同步
4. 回读目标内容或元信息
5. 写入状态和日志

只写出不回读，不算同步完成。

## 11. 错误暴露原则

系统统一错误模型：

- `code`
- `message`
- `detail`
- `source`
- `retryable`

要求：

1. 前端必须直接展示真实错误，不做静默兜底。
2. 外部依赖错误必须标明来源：
   - `fine`
   - `browser`
   - `filesystem`
   - `process`
   - `config`
3. 长任务失败后必须保留完整执行日志。

## 12. 主数据流

统一主链路如下：

1. 浏览器界面发起动作
2. `application` 层编排用例
3. `domain` 层判断规则
4. `adapter` 层执行外部调用
5. 返回统一结果与状态
6. 前端渲染状态和日志

`Codex` 若参与，也必须调用同一套 `application` 用例，不能再走隐藏旁路。

## 13. 测试策略

### 13.1 单元测试

只测业务规则，不依赖真实外部系统，重点覆盖：

- 路径映射
- 配置校验
- 同步决策
- 错误转换

### 13.2 适配器集成测试

分别验证：

- Fine 远程能力
- 浏览器控制
- 文件系统调用
- 进程执行

适配器必须有 mock 验证口和真实环境验证口。

### 13.3 用例测试

重点覆盖完整用例，例如：

- `publish_project`
- `preview_dataset`
- `sync_file`

确保 UI、脚本、AI 共享同一条业务主链路。

### 13.4 E2E 测试

只保留少量关键闭环：

1. 创建项目
2. 探测数据
3. 新建报表
4. 同步发布
5. 预览校验

## 14. 迁移原则

由于本次是直接重建，迁移遵循以下原则：

1. 旧仓仅作为功能参考，不作为新架构的边界来源。
2. 先建立新系统的目录、模块和安装闭环，再逐步迁移能力。
3. 每迁移一项能力，都必须进入新模块边界，不允许直接复制旧逻辑继续堆叠。
4. 若旧实现依赖 `Rust/Tauri/Java` 特定运行时，则优先重构协议和能力接口，不保留旧实现形态。

## 15. 结论

本次重构的核心不是简单换成 `Python + Vue`，而是把现有系统从“按技术栈拼能力”改成“按领域收口能力”。

最终目标是得到一套：

1. 单机本地可运行
2. 浏览器可交互
3. 安装启动可闭环
4. 同步链路唯一
5. 错误可暴露
6. `Codex` 可辅助但不绑死主流程

的 FineReport 开发工具。
